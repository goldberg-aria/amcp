# AMCP Messaging Guide

This document defines safe public messaging for AMCP.

## Core positioning

Primary line:

`AMCP is an open protocol for portable, long-term agent memory.`

Expanded line:

`AMCP helps complete the agent stack: MCP for tools, A2A for agents, and AMCP for memory continuity.`

## What to emphasize

- portable memory
- long-term memory
- continuity across sessions, clients, and runtimes
- compatibility with MCP and A2A
- open protocol plus reference implementations

## What not to imply

- that AMCP replaces MCP
- that AMCP replaces A2A
- that AMCP is already a universal standard
- that all A2A or MCP clients are automatically interoperable with AMCP

## Approved short marketing copy

### One-line version

`MCP gives agents tools. A2A gives them collaboration. AMCP gives them memory.`

### Three-line version

- MCP for tools
- A2A for agents
- AMCP for memory

### Expanded version

`Build agents that access tools through MCP, collaborate through A2A, and remember across sessions through AMCP.`

## Blog/title ideas

- `Completing the Agent Stack: AMCP for Memory Continuity`
- `MCP gave agents tools. A2A gave them collaboration. AMCP gives them memory.`
- `Why Agent Memory Needs an Open Continuity Protocol`

## Outreach guidance

When discussing AMCP with MCP or A2A communities:

- present AMCP as complementary
- keep claims technical and specific
- point to the public spec and reference implementations
- avoid claiming official alignment unless there is explicit confirmation
